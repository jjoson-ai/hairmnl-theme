## Summary

Captured real Chrome CSS Coverage data for `assets/theme.css` and
`assets/custom-theme.css` against five representative templates (home,
brand-collection, product, cart, search). Each rule was bucketed by which
templates use it. The Wave B split should follow the **SAFE strategy**
documented below: ~21–52 KB savings per template on `theme.css`, ~1–11 KB
on `custom-theme.css`.

An **aggressive** strategy could deliver an additional ~190 KB sitewide
savings (the `_unused` bucket) but requires a separate FOUC-verification
pass — recommended as a follow-up bd task, not part of Wave B.

## Methodology

1. **Coverage capture** (`/tmp/ujg6.42/coverage.js`) — Puppeteer + Brave
   (Chromium 126) navigated each template, hydrated JS (scroll-through +
   hover the cart icon + attempt drawer toggle), then stopped CSS Coverage
   and saved byte-range exports.
2. **Rule extraction** (`/tmp/ujg6.42/bucket.py`) — parsed the **CDN-served**
   `theme.css` / `custom-theme.css` (which Shopify whitespace-strips, hence
   byte-aligned with Coverage indices) into discrete CSS rules, including
   `@media`-wrapped child rules with their parent context preserved.
3. **Bucketing** — for each rule, flagged the set of templates whose used-byte
   ranges overlap the rule's source span. The set name becomes the bucket
   (e.g. `home+collection`, or `_universal` / `_unused`).
4. **Selector-identity remap** — because Wave B will split the **disk**
   source files (not CDN-served), the bucket-to-selector mapping
   (`docs/ujg6.42-buckets.json`) is keyed by selector text + media-query
   prefix, not byte offsets.

### Caveat: JS-state false-negatives

Real Coverage measures bytes used during the observed session. CSS rules
that ONLY apply during user interactions we didn't trigger remain in the
`_unused` bucket. Concretely:

- Form validation error states (`.has-error`, `.errors`)
- Drawer / modal open states (the click attempt failed — DOM selectors
  varied across templates)
- Toast/notification states (`.notice`, `.alert`)
- Loading state classes (`.btn-state-loading`, `.btn-state-complete`)
- Some `:focus-visible` rules

The SAFE Wave B strategy below mitigates this by keeping `_unused` in the
core bundle. Removing it would deliver the original 150–200 KB savings
promise — but only after a manual FOUC + interaction-state verification
pass on the dev theme.

## Recommended SAFE emission strategy (Wave B)

### Rule for each rule

```
If bucket == '_universal' OR bucket == '_unused':
    → emit into theme-core.css   (loaded on every template)
Else:
    → emit into theme-{T}.css for EACH template T named in the bucket
    (a rule in `cart+collection+home` goes into all of theme-cart.css,
     theme-collection.css, theme-home.css)
```

(Same rule for `custom-theme.css` → `custom-theme-core.css` + chunks.)

This duplicates 2-and-3-template combos across per-template chunks
intentionally — it keeps the Liquid wiring (Wave C) trivial: one `{% if
template == X %}` per template, no rule-by-rule dedup logic, no shared
"home+collection" file.

The duplication overhead is modest because the multi-template combo
buckets are small (largest is the 4-of-5 `collection+home+product+search`
at 6.5 KB).

## Aggressive strategy (deferred — file as separate bd task)

Removing the `_unused` bucket entirely would deliver:

- theme.css: 186.7 KB sitewide reduction
- custom-theme.css: 38.3 KB sitewide reduction
- **Combined: 225 KB off every pageview**

But this requires verifying that none of the 2,918 `_unused` rules across
the two files are triggered by user interactions our headless Coverage
session didn't reach. A dedicated FOUC + interaction-state verification
pass on dev theme + DevTools Coverage with manual interaction is the
correct path. File as `ujg6.42.5` (or similar) after Wave C lands and the
SAFE split is verified stable.

## Artifacts for Wave B

- `docs/ujg6.42-buckets.json` — keyed by `theme.css` / `custom-theme.css`,
  each value is a map of bucket name → list of rule objects:
  `{ "sel": "...", "bytes": N, "start": N, "end": N }`. The `sel` field
  includes any `@media` parent context as a `"@@@ <media-query> @@@ "`
  prefix for rules nested in `@media`.

- The byte offsets (`start` / `end`) are from the **CDN-served** files
  (whitespace-stripped). They are useful for sanity-checking but NOT
  appropriate for slicing the disk source. Wave B should walk the disk
  source's rules, match by selector identity (selector text + media
  context), and emit the matched rule's exact disk-source text into the
  bucket's destination file.

## Wave B handoff

- The `_universal` and `_unused` buckets together account for 222 KB of
  theme.css and 41 KB of custom-theme.css. These go into
  `assets/theme-core.css` and `assets/custom-theme-core.css` respectively.
- The other buckets go into per-template chunks per the emission rule above.
- After Wave B completes, the original `assets/theme.css` and
  `assets/custom-theme.css` files MUST be preserved as-is on disk (not
  modified, not deleted) so Wave C can revert in one Liquid change if
  FOUC is observed in smoke testing.

## Validation

The Coverage capture script triggered: page load → networkidle2 wait →
viewport scroll (full page in 0.8x-viewport steps) → cart icon hover →
drawer click attempt → 1.5s settle. This caught lazy-mounted carousels
(Swiper, Flickity) and below-the-fold sections but NOT modal/drawer open
states, form validation states, or any state requiring multi-step user
interaction.
