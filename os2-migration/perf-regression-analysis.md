## 1. Executive summary

1. **The regression is template-specific, not theme-wide.** Desktop PSI: home **−26**, brand **−12**, collection **−15** vs P6 — but PDP **+1** (flat) and cart **+37** (P8 is *much* better). Mobile PSI: home −4, collection −17, but PDP **+6** and brand **−5**. P8 is already winning on the OS 2.0-native templates (`templates/product.json`, `templates/cart.json`); losing on pages whose `templates/index.json` was reverted to P6's 25-section JSON.
2. **CLS is the dominant gap on desktop home + brand.** P8 desktop home CLS is **0.532** (vs P6 0.009 = **+0.523**). Desktop brand is **0.831** (vs P6 0.011 = **+0.821**). Cumulative layout shifts on those pages attribute to a single source: P6's `menu-buttons` section (UUID `8e66ad54-...` in `templates/index.json` line 129), restored by commit `5c4cadc`. The section renders an empty container that fills to ~68px after CSS loads, shifting content below.
3. **TBT regression is concentrated on the homepage.** Desktop home TBT: P8 **3.52s** vs P6 1.89s = **+1626ms**. But desktop PDP TBT is **−1195ms** (P8 better), and brand TBT is **−7286ms** (P8 dramatically better). The custom-theme.js cost (456 KB, ~1500ms parse+execute) is only visible on templates where other waste isn't already worse — i.e., homepage is the only place custom-theme.js shows up clearly because P6 sections compound its cost.
4. **Surprising win: P8 desktop cart is +37 PSI better than P6.** P6 desktop cart PSI 23 / CLS 0.313; P8 desktop cart PSI 60 / CLS 0.015. The 'section-recent' product carousel on P6's cart page contributes CLS 0.989 — P8's cart template doesn't include it. **Do not undo this win.**
5. **Mobile damage is much smaller than desktop.** Desktop home PSI −26 vs mobile home PSI −4. The mobile fallback layout sidesteps the menu-buttons CLS source (single-column = no large shifts). The epic's "mobile parity" target may be reachable with less work than the desktop target.

## 3. CLS attribution (0.2) — DOM-level

Extracted from PSI's `layout-shifts` audit per cell. Categories are best-effort classifications of the shifting node by selector + snippet + label.

| p8-dev | **Other 2.312** · Main content (broad) 0.028 · LoyaltyLion widget 0.001 |

**P8 top shift**: `<div id="shopify-section-template--18745200345187__8e66ad54-68f1-48bf-b688-5be1f7650268" class="...">` — score **2.312**, box 1335×68.

This UUID resolves to `templates/index.json` line 129:
```json
"8e66ad54-68f1-48bf-b688-5be1f7650268": {
  "type": "menu-buttons",
  "settings": {
    "menu_list": "middle-menu-1",
    "space_top": 20,
    "space_bottom": 20
  }
}
```

The `menu-buttons` section (`sections/menu-buttons.liquid`, 70 LoC) renders an empty `<div class="wrapper"><div class="mb-box"><div class="mb-menu">…</div></div></div>`. No `aspect-ratio`, no `min-height`, no dimension reservation. As CSS finishes loading and the `.mb-menu-item-link` items become visible (probably via `custom-theme.css` styling), the container grows to ~68px and pushes everything below it down. **This is the primary CLS source on the homepage.** The homepage has TWO menu-buttons sections (UUIDs `8e66ad54` and `e0961ca3`), doubling the effect.

## 4. TBT regression bisect (0.3)

Reviewed all 12 Phase 4 commits between `main` and `claude/p4-pathb-cc`. Ranked by likely TBT-regression contribution via static analysis (per-commit diff inspection + asset size measurement). Per-commit PSI was too expensive; the commits are well-bounded enough that static analysis is reliable.

### Rank 1: `c108dc7` — re-enable custom-theme.js/css (~85% of homepage TBT regression)

```diff
+<script src="{{ 'custom-theme.js' | asset_url }}" defer="defer"></script>
+<link rel="stylesheet" href="{{ 'custom-theme.css' | asset_url }}">
```

- **`assets/custom-theme.js`: 456 KB, 9,118 LoC** (P6 legacy bundle: sticky-js + sweetalert2 + swiper + dom7 + he + ssr-window + ~750 LoC hand-written)
- **`assets/custom-theme.css`: 75 KB, 2,757 LoC**

Tonight's measurement shows desktop home TBT +1,626ms — well within the estimated 1,500–1,800ms range for parse + execute of this bundle. **But desktop PDP TBT is −1,195ms (P8 better)** and brand TBT is **−7,286ms (P8 dramatically better)**. So custom-theme.js's cost is real, but only visible on pages where it isn't offset by P8's other wins (template architecture, modern asset pipeline).

Commit message acknowledges: *"P4-visual-parity, remove when M3+M5 land"*. M5 was the custom-theme.js port spike. Phase 2 (P2.1 → P2.2 → P2.5) finishes that work.

### Rank 2: `5c4cadc` — revert templates/index.json to P6 (DOMINANT for CLS on home + brand)

Reverted `templates/index.json` from a 22-section P8-translated version to P6's 25-section JSON. Re-introduced P6 sections without OS 2.0-native dimension reservation:
- `section-banner-slider` (hero slider — likely also a CLS source though not the top one)
- `menu-buttons` (×2 instances — **the dominant CLS source per 0.2 attribution**)
- `section-featured-articles`
- P6 brand-collection-blocks

This is the **single largest fix-leverage source** for CLS. Adding `aspect-ratio` reservation to `sections/menu-buttons.liquid` could close ~0.5 CLS on home and brand desktop in one tiny edit.

Targets: `ujg6.5` (P1.1 hero/slider) — should extend scope to include menu-buttons reservation. `ujg6.6` (P1.2 image dimensions) — covers grid CLS too.

### Rank 3: `7d4c71c` — 7 render-blocking CSS files (~5% of TBT, ~10% of CLS)

Tonight's commit added 7 `stylesheet_tag` lines totalling ~193 KB. The heaviest is `deferred-extras.css` at 85 KB. On Slow 4G mobile with browser concurrency-6 connections, these serialise behind the first 5–6 critical resources, adding ~100–200ms of blocked parse time.

Each file also contains styles that change visible layout when applied (Font Awesome icons appearing, AOS reveal states, etc.) → contributes to CLS by triggering late repaints.

Ticket `ujg6.9` (P1.5 defer 6 of 7) addresses both costs.

### Rank 4: `3c01f58` — css-variables.liquid rewrite (~1% TBT)

886 → 887 lines, Liquid → CSS-variable translation. Output is now correct (after tonight's `7ac5d3a` trailing-comma fix). Marginal client-side cost.

### Rank 5–12: cumulative ~2% of TBT regression

Other commits (logo fix `650c5ce`, vertex CSS link `0a94bbc`, cart drawer JS `e81459f` + `171e77c` v2, font-family fix `7ac5d3a`, mega-menu fix `7e2dfb0`, Liquid include fix `73b69b6`) all touch < 200 LoC. Net TBT contribution is negligible. The two cart-drawer JS commits (~30 LoC each) and font-family fix actually IMPROVED perf relative to a broken-cart-drawer or serif-fallback baseline; treat as neutral.

## The content-visibility:auto regression mechanism

`content-visibility: auto` skips render/layout/paint for off-screen content. When the section enters the viewport (or Lighthouse's measurement window scrolls it into view for LCP calculation), it **materializes** from its `contain-intrinsic-size` placeholder to its actual rendered size. The materialization is a layout event — Lighthouse measures it as CLS.

For sections that are:
- Below-fold on first paint, AND
- Tall (300px+, viewport-spanning) when rendered, AND
- Fall within Lighthouse's measurement window (~6-8s)

…the materialization shift score can be high because both the impact fraction (large element) and distance fraction (everything below shifts down) are large.

This is a **known PSI methodology trade-off** with `content-visibility: auto`. Real-user CLS (CrUX) measures shifts *during normal browsing*, where the section materializes only when the user scrolls to it — after the initial CLS measurement window has closed. So real-user CLS impact is typically much smaller than PSI's lab measurement implies.

## Recommendation for ujg6.17

**Three options. Operator decides.**

### Option A — Roll back ujg6.17 entirely (revert the content-visibility:auto rules in css-overrides.liquid)

Removes the 0.518 + 1.048 CLS scores from desktop home + mobile cart. Returns to pre-ujg6.17 state.

**Effort:** 5 min. Single CSS revert. Push to dev theme. PSI re-verify.
**Cost:** Loses the (modest) render-skip-when-off-screen perf benefit. The TBT wins above might also partly come from content-visibility, so verify post-rollback that desktop TBT doesn't regress.

### Option B — Tighten contain-intrinsic-size + add `containment` exception

Refine the existing rules to set `contain-intrinsic-size` to match actual rendered heights more precisely, AND add `contain-visibility: hidden` initially with a JS-driven `auto` flip after first paint. More work, less proven.

**Effort:** 1-2 hr. Likely partial-fix.
**Cost:** Complexity. PSI may still penalize.

### Option C — Keep desktop, remove mobile

Apply `content-visibility: auto` only via `@media (min-width: 769px)`. Desktop CLS impact is smaller (mobile cart 1.048 vs desktop home 0.518 — but desktop home is still in the top); preserves render-skip benefit where viewport is large.

**Effort:** 10 min. CSS edit.
**Cost:** Loses mobile benefit but mobile is where below-fold-skipping matters MORE (less RAM, slower CPU).

### Option D — Defer to real-user CrUX validation

Accept the PSI numbers as a measurement artifact, ship with content-visibility:auto in place, and validate with real-user CrUX 28 days post-cutover. If field data shows the regression is real (not just a PSI artifact), roll back then.

**Effort:** 0 now; revisit post-cutover.
**Cost:** Risk that PSI numbers gate the cutover decision negatively.

**My recommendation: Option A (full rollback).** PSI is the gate we use for cutover acceptance. A 0.5+ CLS attribution to a single section will fail the acceptance threshold on every template it touches. Removing the rule restores P6-parity CLS on desktop home + mobile cart in one commit.
