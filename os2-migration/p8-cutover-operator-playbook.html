Phase A — Browser smoke TODAY

Mostly automated. ~30 minutes of script runtime + ≤1 hour of spot-checking anything flagged.

Run the automated suite first

One-time auth capture (~3 min), then run the suite. Cookies stay valid for weeks.

cd scripts/smoke-tests
npm install
npm run install-browsers

# ONE-TIME: capture Shopify admin cookies
mkdir -p .auth
npx playwright codegen --save-storage=.auth/storageState.json https://creations-gdc.myshopify.com/admin
# → log in, visit /?preview_theme_id=141168312419 to verify, close browser

# EVERY RUN: test the dev theme
SHOPIFY_AUTH=.auth/storageState.json npm test
npm run report  # opens the HTML report

54 automated tests run in ~5-8 minutes. The report shows pass/fail per smoke check + screenshots/videos for any failure.

What the automated suite covers

  1. Font CLS — text doesn't jump when fonts load (smoke 1)
  2. Mobile brand-page CLS — brand grid stable on phones (smoke 2)
  3. Per-template JS loading — correct files on correct pages, no 404s (smoke 3)
  4. Reamaze chat defer — pre-interaction zero KB, click + scroll triggers fire (smoke 4)
  5. Collection filter AJAX — no reload + URL update + section fetch (smoke 5)
  6. Lazy-render sections — placeholders swap to content on scroll (smoke 6)
  7. Font preloads + hero — preload links in head + first slide eager+high (smoke 7)
  8. Visual parity vs baseline — 10 full-page + 8 region snapshots × 2 viewports detect any visual drift (smoke 8)

If everything passes, you're done with Phase A. Move to Phase B.

If anything fails, the next slides walk through the spot-check workflow for each smoke check — same content as before but only consult what failed.

First-time baselines: Smoke 8 needs an initial set of baseline screenshots. After verifying the dev theme looks right (one quick eyeball pass), run SHOPIFY_AUTH=.auth/storageState.json npm run update-baselines ONCE, then commit the __screenshots__/ directory. From then on, every npm test detects drift automatically.

Smoke 1 — Font CLS ~5 min

1Open about:blank in a new incognito Chrome window

2Press Cmd/Ctrl+Shift+P → type "throttling" → select Slow 4G

3Paste this URL and press Enter:

https://creations-gdc.myshopify.com?preview_theme_id=141168312419

4Watch the page render. Focus on the homepage headline and body text.

✅ What success looks like

  • Text appears almost immediately in a fallback font
  • When the brand font loads (~1 second later), text might shift a hair sideways — that's OK
  • Nothing should jump vertically. No content should be pushed down the page.

❌ Smells bad if

  • Headings or body text shift down the page when fonts load — that's the bug ujg6.7 fixed; would mean regression
  • Page is blank for >3 seconds with no text at all

Smoke 4 — Reamaze chat defer ~10 min

Reamaze chat used to slow every page load (~250 KB JS). It now loads only when a user interacts. Big win — but we need to verify the chat still WORKS when clicked.

1Fresh incognito window. DevTools open → Network tab → filter reamaze

2Visit homepage:

https://creations-gdc.myshopify.com?preview_theme_id=141168312419

3Confirm: the chat icon (bottom-right) appears within ~1 second of page load, BUT the Network panel shows zero entries for cdn.reamaze.com.

4Click the chat icon. The Reamaze SDK should now load + chat window opens.

✅ Success

  • Chat icon visible from first paint (small, bottom-right)
  • Network panel: 0 KB from cdn.reamaze.com before clicking
  • After clicking: SDK loads + chat window opens normally
  • You can type a message + send it through to Reamaze

❌ Regression

  • cdn.reamaze.com loads on first paint (defer broke — should be 0 KB)
  • Chat icon doesn't appear at all
  • Clicking the icon does nothing / SDK fails to load

Bonus check: same flow on a PDP. And scroll the page past 50% — chat should ALSO load (one of the 5 triggers is "scrolled past 50%").

Smoke 5 — Collection filter AJAX ~10 min

Filtering a collection used to reload the whole page. Now it should swap in just the new grid + update the URL bar.

1Visit a collection with filters:

https://creations-gdc.myshopify.com/collections/best-sellers?preview_theme_id=141168312419

2Find the filter sidebar (left side desktop, drawer on mobile). Tick a filter checkbox.

3Watch the URL bar. It should update with ?filter.v.option1=… WITHOUT the page flashing to white + reloading.

4Click the browser Back button. Should return to the unfiltered grid.

✅ Success

  • Filter checkbox click → grid updates with new products in <1 second
  • URL bar updates to reflect filter
  • No white-flash, no page reload
  • Back button works
  • Direct URL with ?filter.v.… still loads with filter applied (try opening one of the filtered URLs in a fresh tab)

❌ Regression

Page flashes white + reloads on filter click — AJAX broke. Or grid doesn't update. Either way, report.

Smoke 6 — Lazy-render sections ~10 min

Heavy below-fold sections now load only when you scroll near them. Saves ~200 KB on initial page load. Verify it doesn't look janky.

1Open Chrome DevTools → Network tab → filter section_id=

2Visit homepage with the page scrolled to top:

https://creations-gdc.myshopify.com?preview_theme_id=141168312419

3Slowly scroll down. Watch for ?section_id=… requests firing in Network as you scroll near the brand-logo grid + featured-collections sections.

4Visit a PDP and scroll to the bottom — same pattern for the related-products section.

✅ Success

  • Sections appear smoothly as you scroll near them — no visible flicker
  • Brand-logo grid renders with 30+ brand logos visible
  • Featured-collections row renders with the same products as P6
  • PDP related-products section + recently-viewed tabs both work

❌ Regression

  • Empty space where sections should be — placeholder visible but never populates (JS broken)
  • Section content "pops in" jarringly — placeholder dimensions were wrong
  • Brand logos missing or wrong

Bonus: open the Shopify theme editor. The same sections should render normally there (no placeholder visible).

Phase D — Cutover ceremony (the big day) ~4 hours

~30 min of actual cutover work; the rest is monitoring vigilance.

What CC does (technical, ~30 min)

  1. Verify dev theme is in a clean state (no uncommitted changes, pre-cutover-smoke passes)
  2. Capture a pre-cutover snapshot of P6 live (PSI + CrUX) for rollback comparison
  3. Duplicate the P6 theme as "Pipeline 6 - BACKUP" (this is your rollback target)
  4. Publish P8 dev theme as the new live theme
  5. Verify live URL serves P8 (curl + spot-check)

What YOU do (alongside CC, ~3.5 hours)

  1. Immediately post-publish: open https://www.hairmnl.com/ in fresh incognito. Quick smoke: home renders, navigation works, can-add-to-cart, can-view-cart. (~5 min)

  3. First hour: spot-check on a real mobile device. Try a complete checkout flow with a test order.
  4. First 4 hours: monitor real-user analytics for unexpected drops (cart-add rate, page views, bounce rate). GA4 + any RUM tooling.

If anything breaks during the cutover window

Immediate rollback: Shopify admin → Online Store → Themes → click "Actions" → "Publish" on the Pipeline 6 - BACKUP theme. Customers are back on P6 within seconds. Then debrief with CC.

Phase E — Post-cutover monitoring +7 to +56 days

Stay vigilant for 4 weeks. CrUX data validates the migration over 28 days.

First week: daily 15-minute check

Each day, glance at:

  • GA4 — page views, sessions, average engagement, conversion rate. Compare day-over-day; flag any drop >10%.
  • Cart-add → checkout conversion — this is the metric to watch for the mobile-cart regression. If it drops, we hit the worst-case from the cart investigation.
  • Customer support tickets — any new UI complaints? "Can't add to cart", "page won't load", "chat broken" are signals.
  • Search Console — once daily check for Core Web Vitals tab + any new errors.

Emergency rollback — when + how important

P6 backup theme is retained for 4 weeks post-cutover. Use it if needed — no shame.

When to roll back

  • P0: site is broken / checkout fails / customers can't buy
  • P1: cart-add → checkout conversion drops >20% sustained over 2+ hours
  • P1: major visual regression on a primary template (home, PDP) that's customer-visible
  • P2 (judgment call): something subtle is off + you can't get a quick fix from engineering

What's NOT a rollback trigger

  • One bug report from one customer (could be cache / their browser / unrelated)
  • Slightly lower PSI lab score on dev (lab measurements are noisy; field data wins)
  • A minor visual difference on a low-traffic template
  • "Chat icon takes 2 extra seconds to be clickable" (it's deferred by design)

How to roll back

  1. Shopify admin → Online Store → Themes
  2. Find the theme named "Pipeline 6 - BACKUP" (the duplicate CC made on cutover day)
  3. Click Actions → Publish
  4. Confirm. P6 is live within seconds.
  5. Notify CC immediately. We'll diagnose + fix + re-cutover later.

The rollback window is 4 weeks. After that, the backup theme is archived (still recoverable but more involved). Don't put off a rollback decision — if something's broken, switch back, debrief, fix, re-cutover.

What success looks like +28 days

You'll know the migration was successful when:

  • Day-28 CrUX shows P8 LIVE with: mobile LCP < 2500ms · CLS < 0.10 · INP < 200ms on origin level (matching or beating P6's 79%/78%/80% good-rates)
  • Home desktop CLS drops below 0.10 in field data (P6 was failing this at 0.17; P8 dev lab shows 0.009)
  • Home mobile LCP improves below 2500ms field threshold (P6 was 2906ms)
  • Conversion rate unchanged or up versus the pre-cutover baseline week
  • No P0 customer-facing issues filed during week 1
  • Cart-add → checkout conversion stable (the mobile-cart score regression doesn't impact real users)

What we'll do at day 28

Run the full PSI verification matrix on live + capture the CrUX snapshot. Compare against the pre-cutover baseline. Engineer writes a retrospective: what worked, what we'd do differently. File any remaining tickets surfaced.

Then archive the migration program. The store stays on P8 indefinitely.
